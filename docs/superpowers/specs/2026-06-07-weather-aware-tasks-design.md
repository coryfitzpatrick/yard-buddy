# Weather-Aware Task Scheduling with Email Notifications

**Date:** 2026-06-07
**Status:** Approved

## Overview

Tasks generated by AI analysis are currently unscheduled and weather-unaware. This feature adds date windows to every task (a start date and a hard cutoff), keeps those windows current via a daily cron job that re-evaluates them against live weather forecasts, surfaces overdue tasks with clear "do it / skip" guidance, and sends daily email digests to users with urgent or overdue tasks even when they have not opened the app.

## Data Model

### LawnTask changes

Remove the unused `dueDate` field. Replace with:

| Field | Type | Description |
|---|---|---|
| `scheduledStart` | `DateTime?` | Earliest recommended date to act |
| `scheduledEnd` | `DateTime?` | Hard cutoff date |
| `weatherCondition` | `String?` | One of: `no_rain_48h`, `dry_day`, `soil_moist`, `any` |
| `overdueNote` | `String?` | AI-written one-liner for overdue tasks ("Late but still effective" / "Window closed, wait until fall") |
| `stillWorthDoing` | `Boolean?` | `null` = not overdue, `true` = late but act, `false` = skip |

### Yard changes

| Field | Type | Description |
|---|---|---|
| `weatherRefreshedAt` | `DateTime?` | When the cron last recalculated this yard's task windows |

### User changes

| Field | Type | Description |
|---|---|---|
| `notificationsEnabled` | `Boolean` | Default `true`. Opt-out flag. |
| `lastNotifiedAt` | `DateTime?` | Throttle guard to prevent multiple emails per day |

No new tables.

## Task Creation Flow

### Claude prompt changes

Both `generateRecommendations` and `analyzeImages` in `lib/claude.ts` currently pass weather as a one-line string. Both are updated to:

1. Receive the full 5-day forecast from OpenWeatherMap (already fetched, not currently forwarded)
2. Pass a structured forecast block to Claude:

```
Weather forecast (ZIP 30301):
- Today Jun 10: 78F, clear, 5% rain
- Wed Jun 11: 82F, partly cloudy, 15% rain
- Thu Jun 12: 71F, thunderstorms, 90% rain
- Fri Jun 13: 69F, showers, 80% rain
- Sat Jun 14: 75F, clear, 10% rain
```

### Claude output schema additions

Each task in Claude's JSON response gains three new required fields:

```json
{
  "scheduledStartDays": 0,
  "scheduledEndDays": 5,
  "weatherCondition": "no_rain_48h"
}
```

`scheduledStartDays` and `scheduledEndDays` are integers representing days from today. The API route converts these to absolute `DateTime` values before writing to the database.

`weatherCondition` must be one of the four defined values. Claude uses the structured forecast to reason about specific windows (e.g. "apply fungicide before Thursday rain" yields `scheduledStartDays: 0, scheduledEndDays: 2`).

### API route changes

`/api/recommendations` and `/api/analyze` both:
- Pass the full `WeatherData.forecast` array to Claude (not just the summary string)
- Map `scheduledStartDays` / `scheduledEndDays` to absolute dates before `db.lawnTask.createMany`
- Write `weatherCondition` to the new field

## Daily Cron Job

**Location:** `app/api/cron/daily/route.ts`
**Schedule:** Once per day, configured in `vercel.json`
**Auth:** Protected by `CRON_SECRET` env var verified on every request

### Steps (in order)

**1. Fetch yards with pending tasks**
Query all yards that have at least one task with `status = "pending"`. Group by unique ZIP code.

**2. Fetch weather per unique ZIP**
One OpenWeatherMap call per unique ZIP. The free tier (1,000,000 calls/month) covers this comfortably.

**3. Recalculate task windows**
For each pending task, map `weatherCondition` against the fresh forecast to find the next valid window:

| Condition | Logic |
|---|---|
| `no_rain_48h` | Next 2-day stretch with under 30% rain chance on both days |
| `dry_day` | Next single day with under 20% rain chance |
| `soil_moist` | Next window following a day with over 50% rain chance |
| `any` | Shift window forward if past, otherwise leave unchanged |

Update `scheduledStart` and `scheduledEnd`. Update `weatherRefreshedAt` on each yard.

If no valid window is found within the 5-day forecast (e.g. rain every day for `dry_day`), leave the current dates unchanged and do not shift them further into the future. The task will remain in its current window until the next cron run finds a clear window.

**4. Identify and assess newly overdue tasks**
Tasks where `scheduledEnd` is now in the past and `stillWorthDoing` is `null`. For each affected section, make one Claude call (batched per section, not per task) to get `overdueNote` and `stillWorthDoing` for all newly overdue tasks in that section at once.

**5. Send email digests**
Collect users where all of the following are true:
- `notificationsEnabled = true`
- `lastNotifiedAt` is null or before today
- At least one task is overdue with `stillWorthDoing = true`, or at least one pending task has `scheduledStart` on or before today + 3 days

Send one email per user. Update `lastNotifiedAt` after successful send.

## Email Notifications

**Provider:** Resend (free tier: 3,000 emails/month, 100/day)

### Email structure

**Subject (dynamic):**
- "You have [N] overdue lawn task[s] still worth doing" when overdue tasks exist
- "Upcoming lawn tasks for the next few days" when only upcoming tasks

**Body sections (in order):**

1. Overdue tasks still worth doing -- listed first, each with its `overdueNote`
2. Tasks with `scheduledStart` within the next 3 days -- labeled "Coming up soon"
3. Single "View My Tasks" button linking to the dashboard

**Footer:** Unsubscribe link pointing to `/api/notifications/unsubscribe?token=<signed-token>`. The token is an HMAC of the user ID signed with `AUTH_SECRET` so it cannot be forged. The endpoint sets `notificationsEnabled = false` and renders a confirmation page.

### When no email is sent

- User has no overdue (worth-doing) tasks and nothing starting within 3 days
- User already received an email today (`lastNotifiedAt` = today)
- `notificationsEnabled = false`

## Dashboard UI

### Task list structure

Active tasks are grouped into three priority sections: Urgent, High, and Routine. Within each section, tasks are ordered by `scheduledStart` ascending.

Each task card shows:
- Title and description (unchanged)
- Date range badge when both `scheduledStart` and `scheduledEnd` are set:
  - Same month: `Jun 14 - 18`
  - Spanning months: `Jun 28 - Jul 4`
  - Badge turns amber when `scheduledStart` is today or in the past but still within window ("start now")
- Product info (unchanged)

Tasks where `scheduledStart` or `scheduledEnd` is null show no badge, regardless of `weatherCondition`.

### Overdue section

Only shown after the cron has run and assessed overdue tasks (i.e. `stillWorthDoing` is non-null). Tasks where `scheduledEnd` is past but `stillWorthDoing` is still `null` are treated as pending until assessed.

Collapsed by default with a count label ("2 overdue tasks"). Expanding reveals each overdue task with:
- Title
- `overdueNote` as a short line below the title ("Late but still effective" / "Window closed, wait until fall")
- "Do it" and "Skip" action buttons
- When `stillWorthDoing = false`: "Do it" is visually dimmed but still tappable (user can override AI judgment)
- "Skip" sets `status = "skipped"` via the existing PATCH endpoint

### Weather refresh indicator

A single small line above the task list: "Tasks updated Jun 10 at 6:00 AM" pulled from the user's primary yard's `weatherRefreshedAt` (first yard by creation date). Hidden when `weatherRefreshedAt` is null (tasks not yet refreshed by cron).

## Out of Scope

- Per-task notification preferences
- Push notifications
- SMS
- Notification history log
- Multiple emails per day
- User-configurable notification time
