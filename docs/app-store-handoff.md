## What's left — prerequisite path

Numbered in the order you must do them. Steps with `[USER]` need your manual action; steps with `[DEV]` can be dispatched to a subagent once `[USER]` steps for that section are done.

### Phase A — Business setup (LLC + DUNS dependent)

This is the gating phase. Until you have these, you cannot enroll in Apple Developer Program as a business, and Apple Developer is the gate for everything else.

1. **[USER] Get LLC formed** — state-specific. Wyoming, Delaware, or your home state.
2. **[USER] Get EIN from IRS** — free, online, instant. https://www.irs.gov/businesses/small-businesses-self-employed/apply-for-an-employer-identification-number-ein-online
3. **[USER] Get DUNS number from Dun & Bradstreet** — free, takes 30+ business days. Apple requires this for business enrollment. https://www.dnb.com/duns-number/get-a-duns.html
4. **[USER] Decide enrollment path:**
   - **Business enrollment** (requires LLC + DUNS): app published under LLC name. Recommended for liability protection + tax separation. Sets up correctly for any future acquisition.
   - **Individual enrollment** (just your name, no DUNS needed): faster (~1-2 days), $99/year still. App published under your personal name. Acceptable for v1; can migrate to business later.
5. **[USER] Apple Developer Program enrollment** — $99/year. https://developer.apple.com/programs/enroll/ — verification takes 1-2 days (individual) or up to 2 weeks (business, after DUNS).
6. **[USER] Google Play Developer account** — $25 one-time. https://play.google.com/console/signup — verified within hours typically. Can be individual or business; less strict than Apple about DUNS.

**Defer until done:** all subsequent phases.

## Privacy nutrition label declarations (App Store Connect Step 19)

Apple requires you to declare exactly what data the app collects. Per the actual app behavior:

- **Email Address** — Linked to user, used for App Functionality (login)
- **Name** — Linked to user, used for App Functionality (greetings, emails)
- **Coarse Location** (ZIP code only) — Linked to user, used for App Functionality and Personalization (weather, regional lawn care advice)
- **Photos** (lawn photos) — Linked to user, used for App Functionality (AI analysis)
- **Device ID** (FCM push token) — Linked to user, used for App Functionality (push notifications)
- **Usage Data** (interaction analytics via Axiom) — Linked to user, used for Analytics

We do NOT collect: precise location, contacts, browsing history, search history, identifiers (other than the push token), purchases (subscriptions go through Stripe on the web), sensitive info, financial info beyond what Stripe handles.

## Apple compliance reminders (very important)

- **Sign In with Apple parity:** if the app exposes Google sign-in (it does, via NextAuth's Google provider), Apple REQUIRES you to also offer "Sign In with Apple" as an option. Need to add Apple as a NextAuth provider before submission, OR remove Google sign-in from the in-app login screen. Currently the in-app login screen shows whatever the web shows — verify before submission. Quick check: search the `LoginForm.tsx` for "Google" or "OAuth" rendering logic; if visible in-app, add Apple OR conditionally hide Google in-app.
- **No in-app purchase / Stripe steering:** all paywall surfaces hidden per Group 2 + polish. The login screen's "Need an account? Create one at yardanalyzer.com" copy is the safest minimum mention. Don't add anything that says "subscribe," "buy," "upgrade," or directly links to pricing.
- **Permissions ask justifications:** the `NS*UsageDescription` strings in Info.plist are what users see when iOS prompts for permission. They're already set per Task 6.3 — read `ios/App/App/Info.plist` to review them before submission. Apple rejects vague strings like "this app needs access."

## Per-release maintenance (after first launch)

Mobile releases are RARE with the remote-loaded approach — typically 2-4 per year, all driven by platform maintenance (not user features). Web changes propagate instantly without an app update.

What requires an app rebuild + resubmission:
- Capacitor major version bump (~1/year)
- iOS deployment-target bump (post-WWDC each fall, Apple deprecates old targets)
- Android `targetSdkVersion` bump (Aug each year, Play Console enforces)
- Adding/removing native plugins
- Changes to `ios/App/App/Info.plist`, `android/app/src/main/AndroidManifest.xml`, icons, splash
- Critical bug in the native shell itself (rare)

Calendar reminders to set:
- Apple Developer renewal: $99/year, **set reminder 30 days before expiry** — lapsing = apps de-listed within weeks
- Android keystore backup: **on day 1 after generating** — lost keystore = can never update Android app, ever

## Rollback / recovery

If something breaks in production after submission:
1. **Web bug:** fix the web app + deploy via Vercel. Propagates to mobile users on next app open. No app resubmission needed.
2. **Native shell bug:** bump Capacitor version, fix, rebuild, submit to TestFlight first, then production. ~1 day for the fix + 1-7 days Apple review.
3. **DB migration broke something:** Prisma migrations are forward-only; revert via a new migration that undoes the change. Same Vercel-deploy propagation as any web change.
4. **Catastrophic auth bug:** revoke all `BiometricRefreshToken` rows via SQL, force every biometric-opted-in user to re-login with password. App keeps working.
